# §2.7.1 Manual Apply-instruktion

**Migration:** [`supabase/migrations/20260423_f2_7_1_b2b_schema.sql`](../../supabase/migrations/20260423_f2_7_1_b2b_schema.sql)

**Varför manuell:** `supabase db push` är bruten pga migration-drift (hygien-task #25 — 41 KRITISK-tabeller saknar CREATE TABLE i migrations-kedjan). Full `db push`-fix blockeras av Fas 2-utökning §2.1.1. Under tiden: kopiera + kör SQL manuellt i Studio SQL Editor.

---

## Steg 1 — Öppna Supabase Studio

1. Gå till https://supabase.com/dashboard
2. Välj projekt **`urjeijcncsyuletprydy`** (Spickapp's Project)
3. Sidomeny → **SQL Editor** → **New query**

## Steg 2 — Kopiera migration-innehåll

Öppna [`20260423_f2_7_1_b2b_schema.sql`](../../supabase/migrations/20260423_f2_7_1_b2b_schema.sql) i din editor. Kopiera **hela filens innehåll** (från `BEGIN;` till `COMMIT;` — inte verifikations-kommentarerna längst ner).

Klistra in i Studio SQL Editor. Klicka **Run** (eller Ctrl+Enter).

Migrationen är idempotent:
- `IF NOT EXISTS` på alla `CREATE`-statements
- `ADD COLUMN IF NOT EXISTS` på alla `ALTER TABLE`
- `ON CONFLICT DO NOTHING` på `INSERT INTO platform_settings`
- `CREATE OR REPLACE FUNCTION`

**Säkert att köra flera gånger.**

## Steg 3 — Läs DO-blockets NOTICE

I slutet av migration körs verifieringsblocket som `RAISE EXCEPTION` om något saknas, eller `RAISE NOTICE` vid framgång:

```
NOTICE:  OK: 7 B2B-kolumner + b2b_invoice_number_seq + generate_b2b_invoice_number() + company_timezone seed:ade
```

Om du ser detta: migrationen är klar. Om `EXCEPTION` — rapportera felet tillbaka innan §2.7.2 startas.

## Steg 4 — Verifiera manuellt (5 queries)

Kopiera nedan i nytt SQL-fönster och kör. Jämför mot **Förväntat**:

### 4.1 Kolumnerna finns

```sql
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = 'bookings'
  AND column_name IN (
    'business_vat_number', 'business_contact_person', 'business_invoice_email',
    'invoice_address_street', 'invoice_address_city', 'invoice_address_postal_code',
    'invoice_number'
  )
ORDER BY column_name;
```

**Förväntat:** 7 rader, alla `text`, `is_nullable='YES'`.

### 4.2 Sequence finns

```sql
SELECT sequence_name, start_value, increment, minimum_value
FROM information_schema.sequences
WHERE sequence_schema = 'public' AND sequence_name = 'b2b_invoice_number_seq';
```

**Förväntat:** 1 rad, `start_value=1`, `increment=1`, `minimum_value=1`.

### 4.3 RPC returnerar rätt format

```sql
SELECT generate_b2b_invoice_number();
```

**Förväntat första körningen:** `F-2026-00001`.
**Efter varje efterföljande körning:** nästa nummer (`F-2026-00002`, osv.).

> ⚠️ **Notera:** varje `SELECT generate_b2b_invoice_number()` förbrukar ett riktigt serienummer. För B2B-fakturor är det inte ett problem (varje anrop ska ge unikt nummer), men kör INTE queryn mer än ~2-3 gånger manuellt — annars skapas glapp i serien innan första verkliga bokning.

### 4.4 platform_settings har company_timezone

```sql
SELECT key, value FROM platform_settings WHERE key = 'company_timezone';
```

**Förväntat:** 1 rad, `key='company_timezone'`, `value='Europe/Stockholm'`.

### 4.5 Unique index finns

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'bookings' AND indexname = 'idx_bookings_invoice_number_unique';
```

**Förväntat:** 1 rad med `indexdef` innehållande `WHERE (invoice_number IS NOT NULL)`.

## Steg 5 — Verifiera GRANTs (lärdom från §2.5-R2-hygien)

Sequences + RPC kräver explicit GRANT USAGE/EXECUTE för `service_role` — annars kraschar EF-anrop med "permission denied".

```sql
-- 5.1 Sequence GRANTs
SELECT grantee, privilege_type
FROM information_schema.role_usage_grants
WHERE object_schema = 'public'
  AND object_name = 'b2b_invoice_number_seq'
  AND grantee IN ('service_role', 'authenticated');
```

**Förväntat:** 4 rader (2 roller × 2 privilegier: USAGE + SELECT).

```sql
-- 5.2 Function EXECUTE GRANTs
SELECT grantee, privilege_type
FROM information_schema.role_routine_grants
WHERE routine_schema = 'public'
  AND routine_name = 'generate_b2b_invoice_number'
  AND grantee IN ('service_role', 'authenticated');
```

**Förväntat:** 2 rader (2 roller × EXECUTE).

## Om något avviker

- **EXCEPTION i DO-blocket** — migration rollbackad. Kopiera felmeddelandet + rapportera i chat.
- **Rad-count skiljer** från förväntat — kör `4.1` igen. Om fortfarande fel, kontrollera schemat via `\d+ bookings` i psql eller Studio.
- **RPC returnerar fel format** (t.ex. `F-2026-1` utan padding, eller `SF-`-prefix) — allvarligt. Migrationen har troligen `CREATE OR REPLACE`-skrivit över den existerande `generate_invoice_number()`. **Stopp + rapportera.**
  - (Skulle inte hända — migrationen använder helt annat RPC-namn: `generate_b2b_invoice_number`. Men verifiera ändå.)

## Efter lyckad apply

1. Rapportera i chat: "§2.7.1 applicerad, `generate_b2b_invoice_number()` returnerar `F-2026-00001`."
2. §2.7.2 kan starta (UI-form i boka.html).

## Idempotens-test (valfritt)

Kör migrationen **en andra gång** i Studio. Förväntat: ingen error, bara `NOTICE: OK: ...`-rad. Alla `IF NOT EXISTS`/`ON CONFLICT` skyddar mot re-run.
